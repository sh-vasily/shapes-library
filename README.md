# Вопрос №3
Что такое "компилятор", зачем он нужен и почему некоторые языки обходятся без него?

Компилятор предназначен для перевода кода написанного на языке программирования высокого уровня в машинный код, который исполняется процессором. Благодаря компиляции программа работает весьма быстро, однако программа становится зависимой от определенного железа, под который она скомпилирована (в языке Java это обходят благодвря так называемой JVM(java virtual machine), которая преобразует байт код в код,предназначееный для процессора машины, на которой установлена JVM. Кстати говоря, в .net fraemwork используется несколько схожая технология: код написанный на любом языке платформы .net(c#, VB и тд) преобразуется в низкоуровневый IL код. Также компиляторы порой сами оптимизируют код(на счет других других языков точно не скажу, но в с++ такая фишка есть). Собвстенно говоря ответ на последний вопрос(почему некоторые языки обходятся без компилятора) уже содержится в вышеизложенных рассуждениях - аппаратная независимость. Возьмем к примеру JS. Тут даже вопросов не возникает, поскольку язык для построения веб приложений должен быть максимально абстрагирован от "железа". Тоже самое с языком python. Оба языка относятся к так называемым интерпретируемым языкам и для исполнения кода, написанного на этих языка необходим интерпритатор, а не компилятор. Для различие здесь заключается в том, что интерпретатор проходит по покду построчно и ошибка в одной строке не приведет к полной неработоспособности кода, в то время время как в случае с компилятором подобное не пройдет.

# Вопрос №5
Что за приставка "http://" перед адресами сайтов и почему она всё чаще теперь становится "https://"?

Б - безопасность. https означает, что соединение с сайтом проходит через протокол SSL. Буква S после http означает защищенное соединение с использованием SSL - сертификата для доменного имени.

# Вопрос №7
SqlServer, PostgreSQL, SQLLite, MySQL, Oracle, Microsoft Access - разные базы данных с разным функционалом, которые разрабатываются, в основном, разными компаниями с разным видением своего продукта.

Однако все эти базы используют один и тот же язык запросов - SQL, и не планируют от него отказываться. Как так получается? Что такого в SQL, что он подходит всем этим базам?

А если он такой чудесный, то почему многие другие базы данных, вроде MongoDB или Cassandra, его не используют?


Начнем с того, что в принципе единый стандарт необходим, чтобы как-то систематизировать всю работу с данными. Далее, в целом, учитывая достаточно похожие структуры различных диалектов SQL, в теории можно переносить данные из одной БД в другую, внося не так много измениний. Также субд с использованием sql реализуют принципы так называемой реляционной алгебры и реляционного счисляния(они не зависят от конкретной бд). Соответсвенно удобство реляционных бд  заключается в том, что при наличии достаточно большого объема данных, их можно удобно хранить с использованием различных связей. MongoDB и другие NoSQL базы выгоднее использовать, поскольку они банально проще. Связи между таблицами это конечно прекрасно, но только при наличие сложной иерархической структуры системы(например большая организация), в тоже время при хранении, например дынных расчетов, использование связей, ключей и тд является избытычным и приносит лишь неоправданную сложность работы. Соотвественно, исходя из этих рассуждений логично, что производительность NoSQL выше, чем стандартных SQL.

# Вопрос №9
В базе данных MS SQL Server есть продукты и категории. Одному продукту может соответствовать много категорий, в одной категории может быть много продуктов. Напишите SQL запрос для выбора всех пар «Имя продукта – Имя категории». Если у продукта нет категорий, то его имя все равно должно выводиться.

Для реалищации связи "многие ко многим" потребуется вспомогательная таблица.

create table map
(
	product_id int foreign key(product.id),
	category_id int foreign(category.id)
);

select prod.name, categ.name
from product prod left  join map on prod.id = map.product_id
left join category categ on map.category_id = categ.id
